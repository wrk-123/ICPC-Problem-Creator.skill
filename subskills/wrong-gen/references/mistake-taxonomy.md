# 错解与数据设计 taxonomy

这份参考文件用于把“可能出现的错误程序”系统化，而不是靠临场灵感想两个错解。

## 使用方式

拿到题目后，先做一张简短清单：
- 正解思路是什么
- 哪些思路看起来像正解、但实际上是错的
- 哪些程序逻辑上可能正确、但复杂度或实现会错
- 每一类错法需要什么数据才能稳定被卡掉

至少覆盖三层：
- 思路层错误
- 复杂度层错误
- 实现层错误

## 思路层错误

优先检查这些类别：

- 暴力或伪暴力
  - 全枚举
  - 带一点剪枝但本质仍爆炸
  - “先随机几次试试”“启发式调一调”

- 错误贪心
  - 只看局部最优
  - 排序 key 不完整
  - 把必要条件当充分条件
  - 以为交换论证成立，实际不成立

- 错误二分
  - 答案根本不单调
  - 判定函数不是单调的
  - 用错上界、下界
  - `mid` 或循环边界写错，少一位 / 多一位

- 错误 DP
  - 状态缺信息
  - 状态多了一个维度导致炸复杂度
  - 转移漏项
  - 转移顺序错误
  - 滚动数组覆盖顺序错误
  - 把启发式剪枝错当成正确性证明

- 错误图论建模
  - 实际是最短路，结果写成 BFS / SPFA / 暴力松弛
  - 实际应离线处理，结果在线硬算
  - 实际是 DAG / 树结构，结果用过重模型
  - 实际不是网络流，却硬往流上套，复杂度或建模都出问题

- 错误数学建模
  - 公式只在某个前提下成立，却被默认推广
  - 逆元存在条件不满足仍硬用
  - 忽略 `n >= mod`
  - 误把浮点比较当精确判定
    
- 错误的 “人类智慧” 和乱搞
  - 数据跑出结果显示答案呈现某种 “高频固定形态”，选手据此盲目压缩状态空间，且存在明确反例。

## 复杂度层错误

这些错法常常逻辑“几乎对”，但会在系统测试中超时：

- 比正解多一个 `log`
- 每次询问都重建整套预处理
- 每个源点 / 每个状态重新跑一次图算法
- `O(n^2)` 冒充 `O(n log n)`
- `O(n log^2 n)` 冒充 `O(n log n)`
- 只在中小数据可过，大数据近 TL 会死
- 常数极大，理论复杂度没错但实现太慢

典型例子：
- 该用 Dijkstra 却用 SPFA
- 该用 `O(n)` / `O(n log n)` 回文算法却写 `O(n log n)` / `O(n^2)` 方案
- LCA 结构多套一层 log
- 对每个询问做独立 DFS / BFS

## 实现层错误

这些错法可能在样例和小随机上全过：

- 下标 / 边界
  - 0 / 1 下标混淆
  - 区间开闭错误
  - `l < r` / `l <= r` 写错

- 数值
  - `int` 溢出
  - `INF` 太小
  - `mid = (l + r) / 2` 溢出
  - 乘法未转 `long long`

- 多测与初始化
  - 忘记清空数组、vector、visited、lazy tag、DSU
  - 全局状态在不同 case 之间串味

- 顺序
  - 遍历顺序写反
  - 先更新后使用或先使用后更新
  - 滚动数组覆盖原值

- 标准算法误实现
  - Dijkstra：
    - priority queue 取反了，先弹最远点
    - pair 顺序写成 `(node, dist)` 导致按点编号弹
    - 忘记 stale-state 判断，重复扩展同一点
    - `INF` 太小
  - SPFA：
    - 在应为非负边最短路时拿 SPFA 硬做
    - 负环判定 / inqueue / 次数统计写错
  - Tarjan / SCC / 桥边 / 割点：
    - lowlink 更新条件错
    - on-stack 判断错
    - parent edge 处理错
  - LCA / 树剖：
    - 深度对齐写错
    - 区间映射边界错
    - 多一层 log 的实现被大数据卡掉
  - 组合数学：
    - 用 `fact[n] * ifact[k] * ifact[n-k]` 时没有保证逆元存在
    - `n >= mod` 仍按 `mod` 为大质数、小于 `n` 的公式硬算
    - 没考虑 Lucas、prime power、CRT

## 交互题特有错误

- 查询集合选错
- 查询顺序选错
- 提前停止条件错
- fallback 错
- 固定答案策略错
- 问得不够多但自以为已经证明
- 输出合法但没有真正满足目标条件

对“连续扫描”类策略，默认同时检查：
- 从小到大
- 从大到小
- 问满再答
- 遇到第一个“看起来安全”的数就答

## 字符串与哈希

- 用哈希代替严格判定但不做二次确认
- 回文问题写成 `O(n log n)` 或 `O(n^2)`
- KMP / Z / SA / SAM 边界错
- 忽略全相同串、交替串、极短串、极长重复串

## 数学与取模

- 组合数直接用阶乘公式，未检查模数前提
- 逆元只在 `gcd(a, mod) = 1` 时存在，却被默认存在
- `n >= mod` 时仍用普通阶乘逆元方案
- Lucas / prime power / CRT 该上没上
- 浮点二分或几何比较缺少稳健边界

## 数据设计方法

对每一类错法，至少找一个明确的“死亡模式”：

- 错误贪心：构造局部最优会害死全局的极端样例
- 错误二分：构造非单调或边界恰卡转折点的样例
- 错误 DP：构造必须依赖被漏掉状态的信息
- 复杂度错误：构造近上界、卡常数、卡重复更新、卡多次重建
- 实现错误：构造最小 / 最大 / 全相等 / 严格递增 / 严格递减 / 溢出 / 极值权重
- 图算法错误：构造会重复松弛、错误弹堆、错误 `INF`、错误负环处理的数据
- 数学错误：构造 `n >= mod`、模数不互素、答案为 0、边界取整、浮点精度敏感样例

## 测试文件数量

- 单文件单测试：至少 30 个测试文件
- 单文件多测试：至少 20 个测试文件

推荐分布：
- 样例与教学型小数据：少量
- 边界 / 极端结构：一批
- 定向 hack：一批
- 接近时限的大数据：一批
- 纯随机：最后才补

重点不是“凑够数量”，而是保证这些文件之间真的承担不同的卡错职责。
